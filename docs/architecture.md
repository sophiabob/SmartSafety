# Архитектура системы видеоаналитики "SafetyVision"
## 1. Введение
### 1.1. Назначение документа
Данный документ содержит описание архитектуры системы видеоаналитики "SafetyVision", включая модульную структуру, интерфейсы взаимодействия, потоки данных и обоснование принятых решений.
### 1.2. Основание для разработки
Архитектура разработана на основе требований, определенных в документе SRS.md (Лабораторная работа №1). 
## 2. Общая архитектура
### 2.1. Архитектурный стиль
**Модульная архитектура** с четким разделением ответственности между компонентами.
**Обоснование:**
- Слабая связанность модулей
- Возможность независимого тестирования
- Параллельная разработка
- Легкость сопровождения и расширения

## 3. Модули системы
### 3.1. Модуль VideoProcessor (video_processor.py)
| Атрибут | Значение |
|---------|----------|
| **Назначение** | Управление видео-потоком, координация обработки |
| **Ответственность** | Чтение видео, извлечение кадров, наложение разметки, сохранение результата |
| **Основные методы** | `process_video()`, `_draw_annotations()`, `_check_safety_equipment()` |
| **Зависимости** | Detector, ZoneAnalyzer, ReportGenerator, Logger |
### 3.2. Модуль Detector (detector.py)
| Атрибут | Значение |
|---------|----------|
| **Назначение** | Детекция объектов с использованием YOLO |
| **Ответственность** | Загрузка модели, детекция людей, касок, жилетов, фильтрация |
| **Основные методы** | `detect()`, `filter_people()`, `filter_by_class()` |
| **Зависимости** | Ultralytics YOLO |
### 3.3. Модуль ZoneAnalyzer (zone_analyzer.py)
| Атрибут | Значение |
|---------|----------|
| **Назначение** | Анализ опасных зон |
| **Ответственность** | Загрузка зон, проверка попадания человека в зону |
| **Основные методы** | `load_zones()`, `check_violations()`, `get_zones()` |
| **Зависимости** | Конфигурационный файл зон |
### 3.4. Модуль ReportGenerator (report_generator.py)
| Атрибут | Значение |
|---------|----------|
| **Назначение** | Формирование отчетов о нарушениях |
| **Ответственность** | Сбор нарушений, формирование JSON-отчета, экспорт |
| **Основные методы** | `add_violation()`, `generate_report()`, `export_json()` |
| **Зависимости** | Нет |
### 3.5. Модуль Logger (logger.py)
| Атрибут | Значение |
|---------|----------|
| **Назначение** | Логирование событий |
| **Ответственность** | Запись событий с уровнями INFO, WARNING, ERROR |
| **Основные методы** | `log_info()`, `log_warning()`, `log_error()` |
| **Зависимости** | Нет |
### 3.6. Модуль GUI (gui/main_window.py, gui/video_widget.py)
| Атрибут | Значение |
|---------|----------|
| **Назначение** | Пользовательский интерфейс |
| **Ответственность** | Загрузка видео, запуск анализа, отображение результата |
| **Основные методы** | `load_video()`, `start_analysis()`, `on_analysis_finished()` |
| **Зависимости** | PyQt5, VideoProcessor |
## 4. Интерфейсы
### 4.1. Внутренние интерфейсы
| Взаимодействие | Передаваемые данные | Возвращаемые данные |
|----------------|---------------------|---------------------|
| VideoProcessor → Detector | Кадр (numpy-массив) | Список детекций |
| VideoProcessor → ZoneAnalyzer | Координаты людей | Список нарушений |
| VideoProcessor → ReportGenerator | Информация о нарушениях | Статус |
| VideoProcessor → Logger | Сообщение с уровнем | Статус |
| GUI → VideoProcessor | Путь к видео | Статус обработки |
### 4.2. Внешние интерфейсы
| Интерфейс | Направление | Формат |
|-----------|-------------|--------|
| Входной | Загрузка видео | MP4, AVI |
| Выходной | Сохранение результата | MP4 (видео с разметкой) |
| Выходной | Экспорт отчета | JSON, CSV |
| Выходной | Логирование | TXT |
| Пользовательский | GUI | PyQt5 |
## 5. Потоки данных
### 5.1. Поток обработки видео
Входной видеофайл → VideoProcessor (чтение кадров) → Detector (детекция)
→ VideoProcessor (наложение разметки) → Выходной видеофайл
text
### 5.2. Поток анализа нарушений
Detector (детекции) → ZoneAnalyzer (анализ зон) → ReportGenerator (регистрация)
→ JSON-отчет
text
### 5.3. Поток логирования
VideoProcessor, Detector, ZoneAnalyzer, ReportGenerator → Logger → Лог-файл
text

## 7. Соответствие принципам SOLID
| Принцип | Реализация |
|---------|------------|
| **SRP** | Каждый модуль имеет одну ответственность |
| **OCP** | Модули открыты для расширения, закрыты для изменений |
| **LSP** | Объекты заменяемы подтипами без нарушения работы |
| **ISP** | Каждый модуль предоставляет только необходимые методы |
| **DIP** | Зависимость от абстракций, не от реализаций |
